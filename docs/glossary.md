# Glossary

This glossary defines working terminology for the Skills Over MCP documents. It is descriptive rather than normative: the [MCP specification](https://modelcontextprotocol.io/specification/2025-06-18/server/resources) and the [Agent Skills specification](https://agentskills.io/specification) remain the source of truth for their respective formats and protocol surfaces.

| Term | Working definition |
| :--- | :--- |
| Skill | In this working group, a skill is structured workflow context that can be served over MCP, usually as an Agent Skills-compatible directory with a `SKILL.md` file and optional supporting files. This aligns with Agent Skills' directory model; the MCP work adds discovery, transport, URI, and host-loading conventions rather than redefining the skill format. |
| Instruction | In MCP, server instructions are initialization-time guidance a server gives to a client. In skills, instructions are the workflow content inside `SKILL.md` and related files; the two can point to each other, but they have different scope, loading lifecycle, and size constraints. |
| Primitive | An MCP protocol primitive is a named capability surface such as Resources, Tools, or Prompts. In this repository, "primitive" should not mean any useful workflow concept; when skills use existing Resources, "skill" is not a new MCP primitive. |
| Resource | A Resource is MCP's URI-addressed way for servers to expose data or content that clients can incorporate as context. Skills Over MCP uses Resources as the current transport fit because skills are file-like context that can be listed, read, and progressively loaded. |
| Convention | A convention is a documented interoperable pattern that implementations can follow without changing the MCP base protocol. For Skills Over MCP, examples include `skill://` URI structure and `skill://index.json`; these can inform specifications, but they are not themselves new protocol methods. |
| Context | Here, context means information provided to an agent or host to help perform a task, especially workflow know-how exposed as resources. This overlaps with, but is narrower than, "LLM context window": context-as-resources focuses on addressable, selectively loadable material before it becomes model tokens. |
| Progressive disclosure | Progressive disclosure is the staged loading of skill material: lightweight metadata first, full `SKILL.md` when relevant, and supporting files only when needed. This aligns with Agent Skills; in MCP, Resources and resource URIs are the mechanism that lets the host or model read each layer on demand. |
| Control model | The control model describes who decides that context is visible or loaded: the host application, the user, the model, or some combination. MCP Resources are application-driven by default, but Skills Over MCP discussions also consider model-driven resource loading through client-provided helper tools. |
| First-class primitive | A first-class skills primitive would make skills an explicit MCP protocol surface, with dedicated methods, capabilities, and notifications such as `skills/list`, `skills/get`, or skill-list change events. The current resources-based approach intentionally avoids this unless existing primitives prove insufficient. |
| Server author | A server author is the person or organization that ships an MCP server and controls the tools, resources, auth, update cadence, and transport behavior it exposes. A server author may also author skills, but in multi-server or ecosystem scenarios they often provide only one part of the system. |
| Skill author | A skill author is the person or organization that writes the skill's `SKILL.md`, examples, references, and declared assumptions. Under Agent Skills this role owns the skill content; under Skills Over MCP, the same content may be served by a different server author. |
| Discovery | Discovery is the process of learning that a skill exists and getting enough metadata to decide whether it may be relevant. Fetching or reading `SKILL.md` is a separate content-loading step; conflating the two can hide important control, privacy, and performance choices. |
